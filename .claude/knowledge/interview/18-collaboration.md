# 협업 및 갈등 해결

## 답변 프레임워크: STAR

- **S**ituation: 상황/배경
- **T**ask: 해결해야 할 과제
- **A**ction: 내가 취한 행동
- **R**esult: 결과 및 배운 점

---

## 사례 1: 기술 선택 의견 차이

### 예상 질문
"팀원과 기술적 의견이 달랐던 경험이 있나요?"

### Situation
한화솔루션에서 대용량 데이터 처리 방식을 결정해야 했다. 나는 Kafka Batch Consumer + Bulk Insert를 제안했고, 동료는 기존 방식(단건 처리)의 안정성을 선호했다.

### Task
- 팀 내 합의 도출
- 최선의 기술적 결정
- 결정 후 팀 전체가 동의하는 상태

### Action
1. **상대 의견 경청**
   - 기존 방식을 선호하는 이유 파악
   - "안정성"과 "검증된 방식"에 대한 우려 이해

2. **데이터 기반 비교**
   - 두 방식의 성능 벤치마크 수행
   - 결과를 문서화하여 공유

3. **점진적 접근 제안**
   - 한 번에 전면 전환 대신 일부 서비스에 먼저 적용
   - 문제 발생 시 롤백 가능한 구조

4. **의사결정 과정 문서화**
   - ADR(Architecture Decision Record) 작성
   - 선택 이유와 트레이드오프 명시

### Result
- **합의 도출**: 점진적 도입 방식으로 결정
- 첫 서비스에서 **10배 성능 향상** 확인 후 전면 적용
- 동료도 납득하고 함께 구현

### 배운 점
- 데이터가 설득의 핵심이다
- "내가 옳다"보다 "무엇이 최선인가"에 집중
- 점진적 접근이 리스크를 줄인다

## 사례 2: 코드 리뷰 피드백 수용

### 예상 질문
"코드 리뷰에서 어려웠던 경험이 있나요?"

### Situation
42dot에서 PR을 올렸는데, 시니어 개발자가 설계 방식에 대해 강하게 피드백을 주었다. 나는 내 방식이 맞다고 생각했지만, 피드백을 수용해야 할지 고민이 되었다.

### Task
- 피드백의 의도 파악
- 합리적인 결정
- 관계 손상 없이 해결

### Action
1. **감정 분리**
   - "내 코드에 대한 비판"이 아니라 "더 나은 방법 제안"으로 인식

2. **1:1 대화 요청**
   - 텍스트보다 직접 대화
   - "이 피드백의 배경이 궁금합니다"

3. **배경 이해**
   - 과거 비슷한 설계로 문제가 있었다는 것을 알게 됨
   - 경험에서 나온 피드백임을 이해

4. **수정 후 감사 표현**
   - 피드백 반영
   - "덕분에 더 좋아졌습니다" 표현

### Result
- **더 나은 설계**로 개선
- 시니어와 **신뢰 관계 형성**
- 이후 코드 리뷰에서 더 건설적인 토론

### 배운 점
- 피드백은 성장의 기회
- 텍스트보다 대화가 오해를 줄인다
- 경험에서 나온 조언은 경청할 가치가 있다

## 사례 3: 타 팀과의 협업

### 예상 질문
"다른 팀과 협업한 경험이 있나요?"

### Situation
42dot에서 Vehicle Platform API를 개발하는데, 프론트엔드 팀과 API 스펙에 대해 조율이 필요했다. 프론트엔드는 더 많은 데이터를 원했고, 백엔드는 성능을 우려했다.

### Task
- 양측 요구사항 조율
- 합리적인 API 스펙 도출
- 일정 내 개발 완료

### Action
1. **양측 요구사항 명확화**
   - 프론트엔드: "화면에 필요한 데이터를 한 번에"
   - 백엔드: "불필요한 데이터 로딩 최소화"

2. **함께 문서 작성**
   - OpenAPI 스펙을 함께 작성
   - 필드별 필요 여부 논의

3. **기술적 해결책 제시**
   - 필수/선택 필드 분리
   - `expand` 파라미터로 추가 데이터 선택적 로딩

4. **정기 싱크 미팅**
   - 주 1회 30분 API 싱크
   - 변경사항 사전 공유

### Result
- **양측 만족**하는 API 스펙 도출
- `expand` 패턴 **팀 내 표준화**
- 이후 협업이 수월해짐

### 배운 점
- 조기에 스펙을 함께 정의하면 나중에 시간을 절약
- 기술적 해결책이 갈등을 해소할 수 있다
- 정기적 소통이 큰 갈등을 예방

## 사례 4: 주니어 개발자 온보딩

### 예상 질문
"주니어를 도운 경험이 있나요?"

### Situation
한화솔루션에서 신입 개발자가 팀에 합류했다. 프로젝트가 복잡해서 혼자 이해하기 어려운 상황이었다.

### Task
- 빠른 온보딩
- 독립적으로 업무 수행 가능하게
- 팀 생산성 유지

### Action
1. **온보딩 문서 작성**
   - 프로젝트 구조 다이어그램
   - 핵심 컴포넌트 설명
   - 자주 수정하는 파일 목록

2. **페어 프로그래밍**
   - 첫 2주간 함께 작업
   - 실제 업무를 하면서 설명

3. **작은 태스크부터**
   - 버그 수정, 테스트 추가 등
   - 성공 경험 쌓기

4. **질문 환영 분위기**
   - "모르는 건 당연하다" 강조
   - 질문 시간 보장

### Result
- **2주 만에** 독립적 업무 시작
- 온보딩 문서 **팀 표준**으로 정착
- 이후 신입 온보딩 시간 단축

### 배운 점
- 문서화는 투자이다
- 가르치면서 나도 배운다
- 심리적 안전감이 성장을 가속화

## 공통 질문

### Q: 어려운 동료와 일한 경험은?

**답변 구조:**
1. 어떤 점이 어려웠는지 (비난하지 않고)
2. 어떻게 대응했는지
3. 결과
4. 배운 점

**예시:**
"커뮤니케이션 스타일이 달라서 오해가 생기는 경우가 있었습니다. 그래서 중요한 논의는 텍스트 대신 직접 대화하고, 결정 사항은 문서로 정리해서 공유했습니다. 이후 협업이 수월해졌고, 스타일 차이를 존중하는 것이 중요하다는 걸 배웠습니다."

### Q: 리더십을 발휘한 경험은?

**답변 구조:**
1. 어떤 상황이었는지
2. 어떤 역할을 했는지
3. 결과
4. 리더십에 대한 생각

**예시:**
"테스트 문화가 없던 프로젝트에서 Testcontainers 도입을 주도했습니다. 먼저 스스로 예제를 만들고, 팀 세션에서 공유하고, 점진적으로 확산시켰습니다. 직접 명령하기보다 좋은 사례를 보여주는 것이 효과적이라고 생각합니다."

### Q: 피드백을 주고받는 스타일은?

**답변:**
"피드백을 줄 때는 구체적이고 건설적으로, 예를 들어 '이 부분을 이렇게 바꾸면 이런 장점이 있어요'라고 합니다. 받을 때는 감정보다 내용에 집중하려고 합니다. 잘 이해되지 않으면 바로 물어봅니다."

## 답변 팁

1. **부정적으로 말하지 않기**
   - "그 팀이 문제였다" ❌
   - "의사소통 방식이 달라서 조율이 필요했다" ✅

2. **결과보다 과정 강조**
   - 무엇을 했는지가 중요
   - 과정에서 배운 점 포함

3. **나의 역할 명확히**
   - "팀에서 했다" ❌
   - "제가 문서를 작성하고 공유했다" ✅

4. **배운 점으로 마무리**
   - 성장 마인드셋 보여주기
