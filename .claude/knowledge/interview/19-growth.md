# 성장 및 학습 경험

## 답변 프레임워크: STAR

- **S**ituation: 상황/배경
- **T**ask: 해결해야 할 과제
- **A**ction: 내가 취한 행동
- **R**esult: 결과 및 배운 점

---

## 사례 1: 우아한테크캠프 Pro 우수 수료

### 예상 질문
"자기 개발을 위해 노력한 경험이 있나요?"

### Situation
한화솔루션 재직 중, 실무에서 더 좋은 코드를 작성하고 싶었다. 회사 업무만으로는 성장에 한계를 느꼈다.

### Task
- 외부 교육을 통한 역량 강화
- 업무와 병행 가능한 학습
- 실무에 적용 가능한 내용 학습

### Action
1. **우아한테크캠프 Pro 5기 지원**
   - 경쟁률이 높은 과정에 도전
   - 업무 외 시간 활용 계획

2. **10주간 미션 수행**
   - 주 1회 미션 제출
   - 코드 리뷰 피드백 반영

3. **학습 내용 정리**
   - 미션별 회고 작성
   - 실무 적용 포인트 메모

4. **실무에 적용**
   - 객체지향 설계 원칙
   - 테스트 작성 습관

### Result
- **우수 수료** 달성
- 코드 품질 향상
- 테스트 작성 습관 형성

### 배운 점
- 외부 시선이 성장을 가속화
- 작은 미션이라도 꾸준히 하면 큰 변화
- 학습과 실무 적용은 별개가 아니다

## 사례 2: Spring Camp 2023 발표

### 예상 질문
"지식 공유 경험이 있나요?"

### Situation
한화솔루션에서 글로벌 서비스를 개발하면서 Timezone/DST 관련 많은 삽질을 했다. 다른 개발자들도 같은 문제를 겪을 것 같았다.

### Task
- 경험 정리 및 공유
- 발표 준비
- 커뮤니티 기여

### Action
1. **경험 정리**
   - 겪었던 문제와 해결책 문서화
   - 실무 예제 코드 준비

2. **Spring Camp 신청**
   - CFP(Call for Proposals) 작성
   - 발표 주제 선정

3. **발표 준비**
   - 슬라이드 제작
   - 리허설 수차례

4. **발표 및 후속**
   - 현장 질의응답
   - 블로그 포스팅

### Result
- **Spring Camp 2023** 발표
- 커뮤니티 피드백
- 개인 브랜딩 강화

### 배운 점
- 가르치면서 더 깊이 이해
- 발표는 정리의 기회
- 경험 공유는 모두에게 도움이 된다

## 사례 3: 오픈소스 기여

### 예상 질문
"오픈소스에 기여한 경험이 있나요?"

### Situation
42dot에서 OpenFeign QueryDSL을 사용하다가 버그를 발견했다. `@JdbcTypeCode` 어노테이션이 적용된 필드가 KSP 코드 생성에서 인식되지 않았다.

### Task
- 버그 원인 파악
- 수정 및 PR 제출
- 메인테이너와 소통

### Action
1. **이슈 분석**
   - 소스 코드 분석
   - 재현 케이스 작성

2. **수정 구현**
   - 코드 수정
   - 테스트 케이스 추가

3. **PR 제출**
   - 변경 내용 설명
   - 메인테이너 피드백 반영

4. **머지 완료**
   - 코드 리뷰 대응
   - 머지 확인

### Result
- **PR #1127 머지**
- 실제 프로젝트에서 버그 해결
- 오픈소스 기여 경험

### 배운 점
- 오픈소스 기여는 생각보다 어렵지 않다
- 작은 버그 수정도 가치 있다
- 메인테이너와 소통하는 법을 배웠다

## 사례 4: IntelliJ 플러그인 개발

### 예상 질문
"사이드 프로젝트 경험이 있나요?"

### Situation
개발하면서 의존성 정보를 확인하려면 매번 Maven Repository 사이트를 방문해야 했다. 불편함을 해결하고 싶었다.

### Task
- IntelliJ 플러그인 개발
- JetBrains Marketplace 배포
- 지속적 유지보수

### Action
1. **아이디어 구체화**
   - 필요 기능 정의
   - 유사 플러그인 조사

2. **개발**
   - IntelliJ Platform SDK 학습
   - 플러그인 구현

3. **배포**
   - JetBrains Marketplace 등록
   - 문서 작성

4. **유지보수**
   - 사용자 피드백 반영
   - 버그 수정

### Result
- **Gradle/Maven Dependency Explorer** 출시
- JetBrains Marketplace 등록
- 실제 사용자 확보

### 배운 점
- 내가 필요한 것이 다른 사람도 필요하다
- 작은 불편함도 해결할 가치가 있다
- 사이드 프로젝트로 새로운 기술을 배울 수 있다

## 강점과 약점

### Q: 본인의 강점은?

**답변:**
"문제를 구조화하고 문서화하는 것이 강점입니다. 한화솔루션에서 DLT 패턴을 도입할 때, 문제 분석부터 해결책, 모니터링까지 문서로 정리해서 팀에 공유했습니다. 이 문서가 이후 팀 표준이 되었고, 신규 입사자 온보딩에도 활용되었습니다."

### Q: 본인의 약점은?

**답변:**
"완벽주의 성향이 있어서 때로는 빠른 배포보다 완성도에 집착할 때가 있습니다. 이를 개선하기 위해 '일단 배포하고 개선하자'는 마인드셋을 연습하고 있습니다. Feature Flag를 활용해 작은 단위로 배포하고, 실제 피드백을 받아 개선하는 방식을 적용하고 있습니다."
